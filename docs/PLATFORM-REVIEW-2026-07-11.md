## Executive Summary

The platform is genuinely capable and, in places, unusually disciplined for a
private-operator tool: dependency-injected services, real audit/permission plumbing on
sensitive POSTs, routing-proof/trace metadata on every routed call, and a Carbon-inspired
console. It is held back by a small number of **systemic weaknesses concentrated in the
highest-value files**: the two runtime entry points (the 2,200-line proxy and the
1,800-line console) are the least tested (19% and 15% line coverage), the governance-locked
model registry was written non-atomically, several cost-bearing and terminal-exposing
surfaces were under-authorized on a `0.0.0.0`-bound console, and the packaged install path
did not actually ship the code it needs to run.

**Ground truth (verified by running it):** 219 baseline unit tests pass; `py_compile`
is clean across all modules; the console boots and serves `/health` and `/version`; but
the two entry points carry almost no test coverage and the release coverage gate was
decorative (`--fail-under 1`). A live operator console and proxy were found already running
on this host (`0.0.0.0:18181` / `127.0.0.1:18081`); the reviewer did not disturb them.

**Totals:** 116 findings — **8 P0**, 38 P1, 53 P2, 17 P3. By theme: 32 security,
29 reliability, 16 performance, 8 critical-defect, plus architecture, UX, consistency,
accessibility, and quick wins.

### The systemic threads

1. **The source of truth was fragile.** `config/models.json` was rewritten with a bare
   truncate-then-write on hot read paths from multiple threads; a torn read fell back to
   bundled defaults that were then saved back, silently wiping operator state. *(Fixed.)*
2. **Cost-bearing actions had gaps.** The proxy image endpoint skipped budget + allowlist;
   the console did not require `model_use` for chat/image or `tmux_control` for
   terminal *reads*; the Dedicated build was not idempotent, so a double-click could
   orphan a billing GPU. *(All fixed.)*
3. **A `0.0.0.0` console exposes a shell.** The WebSocket PTY bridge authorized on
   "any token," not `tmux_control`, and was never audit-logged; `/api/agentboard` leaked
   live pane contents. *(Fixed.)* Wallpaper SSRF and a stored-XSS vector in template
   rendering remain as tracked P1s.
4. **The packaged product was dead on arrival.** `install.sh` and the RPM spec never
   copied `src/`, `templates/`, or `config/`, so the installed console could not import
   and the launcher/proxy fell back off the governed registry. *(Fixed in the scripts;
   needs a real packaged-install acceptance test — see `docs/NEEDS-OPERATOR.md`.)*
5. **Observability grows without bound.** `usage.jsonl` / `traces.jsonl` / proxy log are
   re-parsed in full on hot paths with no rotation; the frontend fires ~10 heavy endpoints
   on load regardless of active tab. *(Tracked; highest-leverage remaining performance work.)*
6. **Streaming is simulated.** The proxy fully buffers each upstream response before
   replaying it as one SSE burst, so first-token latency equals total latency for every
   Claude Code request. *(Tracked P1 — the biggest interactive-quality win.)*

## Detailed Findings

Every finding lists the affected file/line, the problem, why it matters, the recommended

### Critical Defects

- ✅ FIXED · **P0 — install.sh and the RPM spec install a broken product: src/, templates/, and config/ are never installed** — `install/install.sh`:112 · effort medium · fix-risk low
  - *Problem:* install.sh (lines 109-133) and matts-value-set.spec (lines 50-66) copy only claude-DO.sh, do-anthropic-proxy.py, image-studio.py, matts-console.py, the wrappers, and matts-image. But image-studio.py hard-imports `src.console.*` (image-studio.py lines 25-40) and serves pages from a `templates/` dir resolved relative to the script (image-studio.py line 1393), and both the launcher and proxy default the model registry to `<script_dir>/config/models.json` (claude-DO.sh line 25; do-anthropic-proxy.py lines 95-99). None of src/, templates/, or config/ are installed.
  - *Why it matters:* The installed `matts-console` crashes immediately with ModuleNotFoundError; the installed `claude-do` and proxy silently fall back to the embedded hardcoded model list, directly violating the GOVERNANCE lock that config/models.json is the model source of truth. The whole packaged deployment path is dead-on-arrival, and neither test-install.sh nor release-check.sh would catch it (test-install.sh only checks file existence of the four scripts).
  - *Recommendation:* Install src/, templates/, and config/ (models.json as %config(noreplace) or bootstrap from default-models.json into the data dir) alongside the scripts in both install.sh and the spec; add a post-install smoke step that actually runs `matts-console --help`/imports image-studio and `claude-do --list-models` against the installed tree; extend test-install.sh to assert those directories exist.
- ✅ FIXED · **P0 — build()/resume are not idempotent: no guard against an already-provisioned server, so a double build orphans a billing DigitalOcean resource** — `src/console/services/dedicated.py`:689 · effort low · fix-risk low
  - *Problem:* build() never checks the current state or existing inference_id before POSTing /v2/dedicated-inferences (dedicated.py:689-777). /api/dedicated/resume routes to the same dedicated_build (api_handler.py:256-258). If the operator clicks Build twice (double-click, retry after a slow response, or build + resume), or two sessions build concurrently, DigitalOcean creates two cost-bearing dedicated servers; cfg['inference_id'] is overwritten by the second response (dedicated.py:766), so the first server becomes an untracked orphan that bills hourly with no local record, no teardown path, and no budget accounting (runtime_cost_summary only models one lifecycle). There is also no lock around the whole load-config → preflight → POST → save sequence, so two concurrent requests both pass preflight and both fire the create call.
  - *Why it matters:* This is a cost-bearing cloud action that can fire twice — exactly what GOVERNANCE.md's cost-safety section forbids. An orphaned GPU server can bill hundreds of dollars before anyone notices, and nothing in the console will ever show it.
  - *Recommendation:* At the top of build(): if cfg state is in {new, creating, provisioning, active, idle_warning} or inference_id is set, refuse with 409 and require explicit teardown/rebuild confirmation (a 'rebuild: true' flag that tears down the old id first). Guard the whole build sequence with a lock. Optionally list /v2/dedicated-inferences during preflight and warn about unknown live instances.
- ✅ FIXED · **P0 — Model registry (config/models.json) has no atomic writes, no locking, and is unconditionally rewritten on read paths — corruption can silently reset it to bundled defaults** — `src/console/services/model_registry.py`:152 · effort medium · fix-risk low
  - *Problem:* save() does a plain path.write_text() (truncate-then-write) with no temp-file/os.replace and no lock. The console runs a ThreadingHTTPServer plus a background dedicated-policy worker, and multiple hot paths rewrite the file concurrently: sync_serverless_model_catalog() calls save_model_registry(merged) unconditionally even when nothing changed (serverless_catalog.py:342), and it is triggered by every GET /api/models, GET /api/status, and GET /api/dedicated/status via models_payload(refresh_catalog=True) (image-studio.py:1052-1053, api_handler.py:92-93/137-138/163). register_model() also saves unconditionally on every dedicated status poll (dedicated.py:562, called from status_payload at dedicated.py:663-664). Worse, load_with_status() treats any parse failure — including reading a half-written file from a concurrent writer — as 'defaults_after_error' and returns the bundled DEFAULT registry (model_registry.py:127-139); the very next sync or register_model then SAVES that defaults-based list, permanently wiping the operator's registry (manual enables/disables, aliases, access-audit results, dedicated entries). Long-running writers amplify this: audit_model_access_key() loads a snapshot, probes every serverless text model serially over the network (30s timeout each, serverless_catalog.py:237-273), then saves the stale snapshot minutes later, clobbering anything written in between.
  - *Why it matters:* config/models.json is the governance-locked source of truth for the entire platform (selectors, /v1/models, routing, pricing, Dedicated state). A torn write or lost update corrupts or silently resets it, which is data loss of the system's most important file, and the failure mode is invisible (the app keeps running on defaults).
  - *Recommendation:* 1) Make save() atomic: write to a temp file in the same directory and os.replace() onto the target. 2) Serialize all registry read-modify-write cycles behind a module-level threading.Lock (or fcntl.flock for multi-process safety with the proxy). 3) Stop saving when nothing changed: compare merged output to loaded input in sync_serverless_model_catalog and register_model before writing. 4) Never persist the defaults fallback: if load reports source=defaults_after_error, refuse to save and surface the error; keep a .bak of the last-good file before any overwrite.
- ✅ FIXED · **P1 — Image generation endpoint bypasses budget enforcement and the model allowlist entirely** — `do-anthropic-proxy.py`:1545 · effort low · fix-risk low
  - *Problem:* _budget_error is invoked only on the /v1/messages path (line 1703). The /v1/images/generations handler (lines 1545-1696) performs rate-limit, cache, and circuit checks but never calls _budget_error, and never checks `model not in self.server.models` (the chat path does at line 1741). Image costs ARE written to cost_file (line 1680), so images consume the budget that only chat requests respect, while image requests themselves can never be blocked — and any arbitrary model id is forwarded upstream with the operator's token.
  - *Why it matters:* Governance locks budget-guard behavior as a cost-safety control. At $0.08/image, an unattended loop (or a runaway console feature) can spend without bound even after daily/monthly budgets are exhausted, and disabled/forbidden registry models remain invocable through this route, violating the config/models.json source-of-truth lock.
  - *Recommendation:* At the top of the images handler, run the same _budget_error check (return 402 with trace) and reject models not present in server.models / not type image, mirroring the chat path's 404 policy-decision response.
- **P1 — matts-image ignores an existing token file and always demands the env-key override** — `matts-image`:34 · effort low · fix-risk low
  - *Problem:* write_token() (matts-image lines 34-41, called unconditionally in main() at line 147) exits with an error whenever access_key() is empty — and access_key() only returns the env override (gated on MATTS_VALUE_SET_ALLOW_KEY_OVERRIDE=1) or the empty EMBEDDED_ACCESS_KEY. Unlike claude-DO.sh lines 11-15/220-225, it never checks whether the token file already contains a key. So on a machine already configured through claude-DO.sh, `matts-image "prompt"` always fails with 'Set MATTS_VALUE_SET_ACCESS_KEY...'. When the override IS set, both matts-image (line 40) and claude-DO.sh (line 221) persist the override key into the token file, contradicting CLAUDE.md's documented 'one-run' override and clobbering the stored durable key. matts-image also spawns the proxy with a hardcoded model list and no --model-config-file (lines 84-99), so a later claude-DO.sh run detects the capabilities mismatch and kills/restarts that proxy.
  - *Why it matters:* The documented image-generation workflow ('matts-image "your prompt"' in install.sh's own quick start) is broken on any normally configured host; the 'one-run' key override is actually a destructive persistent overwrite of the operator's stored credential; and the two launchers fight over the shared proxy because they disagree on the model registry.
  - *Recommendation:* In matts-image, mirror claude-DO.sh: use the existing non-empty token file when no override is present, and only write the file when an explicit override is given (or better, keep the override in memory and pass it via --token-file to a temp path so 'one-run' is truthful in both tools). Start the proxy with --model-config-file and the registry-derived model list (reuse the same registry loader).
- ✅ FIXED · **P1 — Keep-alive extension can SHORTEN the teardown deadline: expiry forces teardown_due even when normal idle policy would allow much more time** — `src/console/services/dedicated.py`:285 · effort low · fix-risk low
  - *Problem:* idle_policy_state() computes teardown_due = active and ((idle >= teardown_seconds and not extension_active_unused) or extension_expired_unused) (dedicated.py:282-285). extension_expired_unused is true as soon as now >= keep_alive_until with no work since the extension started — regardless of how much normal idle budget remains. Concrete failure: idle_teardown_seconds=3600; operator finishes work at T, clicks 'keep alive 300s' at T+100 (keep_alive_until=T+400). At T+400 the policy worker's enforce_policy() sees extension_expired_unused=true, marks teardown_due, and destroys the server (dedicated.py:353-362) — 3200 seconds earlier than the configured policy, with reason 'keep_alive_extension_expired'. An action labeled and audit-logged as 'Dedicated idle teardown extended' (dedicated.py:394) actually accelerates teardown whenever keep_alive_until < last_work_at + idle_teardown_seconds.
  - *Why it matters:* A cost-bearing resource is destroyed contrary to configured policy and operator intent; an operator who taps keep-alive 'just to be safe' mid-session gets their GPU server killed under them. It also makes the keep-alive control untrustworthy.
  - *Recommendation:* Treat keep_alive_until as a floor, not a fuse: teardown_due should be (idle >= teardown_seconds AND now >= keep_alive_until [if an unused extension exists]) — i.e., after expiry, fall back to the normal idle computation rather than forcing teardown. Update the countdown to max(normal_countdown, keep_alive_until - now).
- ✅ FIXED · **P1 — Dark mode replaces Create's rotating wallpaper with a hardcoded remote Unsplash photo** — `templates/main.html`:132 · effort low · fix-risk low
  - *Problem:* The light-mode `.create-view:before` was later overridden to gradients-only (line 174) so the JS wallpaper stage (`.create-wallpaper-stage`, z-index 0) shows through the z-index-1 overlay. But `body.dark .create-view:before` (line 132) is more specific and still declares `url("https://images.unsplash.com/photo-1500530855697...")` as an opaque cover background. In dark mode the :before therefore fully covers the wallpaper stage: the rotating wallpaper, mood filters (`[data-mood]` rules target .create-wallpaper only), and the Refresh button all silently stop having any visible effect, and the page fetches from an external CDN.
  - *Why it matters:* Violates two Interface Rules at once: "Create should remain wallpaper-forward" and "Dark mode ... must work across Code, Create, and Console." It also adds an undeclared external network dependency to a private-operator tool — offline, dark-mode Create degrades to a flat gradient with a broken-looking Refresh control.
  - *Recommendation:* Change body.dark .create-view:before to gradients only (mirror line 174 with darker stops) and delete both Unsplash url() declarations (lines 32 and 132). Verify with browser smoke in dark mode that wallpaper rotation and mood filtering are visible, per the UI-evidence rule.
- **P2 — Cost-ledger writes fail silently, which silently disables budget enforcement** — `do-anthropic-proxy.py`:853 · effort low · fix-risk low
  - *Problem:* _write_jsonl (lines 846-854) swallows all OSError with `pass`. If ~/.cache/matts-value-set/ is unwritable (permissions, disk full, read-only mount), every usage record is dropped with no log line, no stderr, no health flag. _budget_error then computes totals from a stale/empty file and never trips. Additionally, records larger than the 8KB stdio buffer can be split across multiple write() syscalls, so concurrent appenders (proxy threads plus the console process writing the same traces.jsonl) can interleave and corrupt lines.
  - *Why it matters:* Budget enforcement is a governance-locked cost-safety control, and its failure mode here is invisible: spending continues while the ledger silently stops growing. Corrupted JSONL lines are skipped by _usage_totals (line 1264), silently under-counting spend the same way.
  - *Recommendation:* On OSError, at minimum print a loud one-time warning and set a server flag surfaced in /v1/claude-do/capabilities and /budget ('cost ledger unwritable'); consider failing closed (reject requests) when the cost file is unwritable and a budget is configured. Serialize writes per-path with a lock and write each record with a single os.write on an O_APPEND fd.

### Security Risks

- ✅ FIXED · **P0 — Launcher silently defaults every session to --dangerously-skip-permissions** — `claude-DO.sh`:470 · effort low · fix-risk low
  - *Problem:* In claude-DO.sh lines 466-472, when the caller does not pass a permission flag, non-root users get `claude_args+=(--dangerously-skip-permissions)` appended automatically (root/sudo gets acceptEdits). Every wrapper (claude-deepseek, claude-glm, ...) and every tmux terminal the web console spawns therefore runs Claude Code with ALL permission prompts disabled, with no warning printed.
  - *Why it matters:* GOVERNANCE.md requires sensitive actions to be permission checked. This default hands an autonomous agent unrestricted file-write and command execution on the host. Combined with the console exposing these tmux terminals on 0.0.0.0:18181, a single leaked console token yields a browser-driven, permission-free agent on the machine. It also silently changes the safety posture for an operator who never asked for it.
  - *Recommendation:* Never inject --dangerously-skip-permissions implicitly. Default to Claude Code's normal permission mode (or acceptEdits at most), gate skip-permissions behind an explicit opt-in (e.g. MATTS_VALUE_SET_SKIP_PERMISSIONS=1 or a --yolo flag), and echo a prominent warning to stderr whenever it is engaged. Document the change in SECURITY.md.
- ✅ FIXED · **P0 — WebSocket terminal bridge grants full shell to any authenticated identity (viewer/billing) and is never audit-logged** — `src/console/handlers/websocket_handler.py`:68 · effort low · fix-risk low
  - *Problem:* do_GET routes /ws/tmux to do_websocket_tmux() (image-studio.py:1647) which constructs TmuxWebSocketHandler(self.authorized). handle() gates only on self.authorized() (websocket_handler.py:68), which in auth_handler.authorized() returns True for ANY identity with a non-empty permission set. A 'viewer' token (permissions {view_console,view_traces,view_billing}) or 'billing_admin' token therefore passes, and the handler then execvp's `tmux attach-session -t <name>` into a PTY bridged bidirectionally to the socket (websocket_handler.py:92,146). Every POST terminal path (/api/tmux/send, /api/terminal/write) is gated on the tmux_control permission in SENSITIVE_POST_PERMISSIONS, but the strictly-more-powerful live read/write shell bridge is not. No append_audit call is made for the attach or for any keystroke written through it, so SECURITY.md's claim that 'tmux terminal writes are audited' is false for the WS path.
  - *Why it matters:* This is a vertical privilege escalation to arbitrary command execution: a read-only scoped role token (the exact feature SECURITY.md promotes for delegated/monitored access) obtains an interactive shell as the console's OS user, with zero audit trail. On a 0.0.0.0-bound console any LAN holder of a low-privilege token gets host RCE. It directly violates the governance lock that sensitive/terminal actions must be permission-checked and audit-logged.
  - *Recommendation:* In TmuxWebSocketHandler.handle(), resolve the caller's identity and require has_permission(identity, tmux_control-equivalent) before upgrading; return 403 otherwise. Pass the identity (not just a boolean) into the handler. Emit an append_audit('tmux.attach', actor=..., permission='tmux_control') on connect and on disconnect, and consider rate-limiting /ws/tmux (it currently skips rate limiting because the path is not under /api/).
- ✅ FIXED · **P1 — Auth enforcement on the live HTTP server is never tested — every HTTP-level test disables auth** — `image-studio.py`:1746 · effort medium · fix-risk low
  - *Problem:* The only tests that exercise StudioHandler over real HTTP (tests/test_console_smoke.py ApiVersionHttpSmokeTests and ApiRateLimitHttpSmokeTests) run with patch.object(studio, 'auth_enabled', return_value=False). AuthHandler is unit-tested in isolation (tests/test_auth_handler.py), but the wiring that actually enforces it — do_GET L1667 (401/login), do_POST L1746 (auth), L1755-1760 (permission check + denied audit), L1770-1782 (allowed/completed audit records), and the pre-auth /api/auth/refresh carve-out at L1737 (which also bypasses rate limiting since check_rate_limit runs later at L1751) — has zero coverage. Deleting the has_permission block at L1758 or reordering read_json before authorized() would pass the entire suite and CI.
  - *Why it matters:* This console binds 0.0.0.0:18181 and exposes local terminals; token auth is the only barrier. The audit trail for sensitive actions (a governance lock) is produced exactly in these untested lines. The unrate-limited pre-auth refresh endpoint is also an untested brute-force surface for refresh tokens.
  - *Recommendation:* Extend tests/test_console_smoke.py with an auth-enabled HTTP suite (patch auth_token to a fixed value, role_tokens to include a viewer profile, and redirect audit/trace files to temp paths): assert (a) GET /api/models without token → 401 and '/' returns the login page; (b) owner token via query/header/bearer → 200; (c) viewer token POST /api/dedicated/build → 403 and a denied audit record written; (d) owner POST to a sensitive route writes allowed+completed audit records with redacted body; (e) POST /api/auth/refresh works pre-auth with a valid refresh token and is rejected otherwise; add rate limiting to the refresh route or a test documenting the exemption.
- ✅ FIXED · **P1 — Registry-mutating, cost-incurring catalog sync exposed as GET, bypassing the sensitive-permission map** — `src/console/handlers/api_handler.py`:104 · effort low · fix-risk low
  - *Problem:* GET /api/models/serverless-catalog calls sync_serverless_model_catalog(force=True, validate_access=True) — it adds/updates registry models, disables unauthorized ones, and runs paid access probes. - ✅ FIXED · **P1 — State-mutating GET /api/models/serverless-catalog rewrites the source-of-truth registry and syncs the proxy with no permission check or audit** — `src/console/handlers/api_handler.py`:104 · effort low · fix-risk medium
  - *Problem:* The GET branch calls sync_serverless_model_catalog(force=True, validate_access=True) then proxy_sync_payload(force=True), mutating config/models.json (the locked source of truth) and reconfiguring the proxy. do_GET never calls permission_for (auth_handler.permission_for returns None for anything but POST, auth_handler.py:102-105) and never append_audit's. So any principal with any permission triggers a full external DigitalOcean catalog import + registry rewrite + proxy sync. The semantically-equivalent POST /api/proxy/sync requires model_admin + proxy.sync and is audited.
  - *Why it matters:* GOVERNANCE explicitly locks: model-registry actions must be permission-checked and audit-logged, and config/models.json is the source of truth. A mutating GET is also CSRF-shaped (triggerable by a simple navigation/img/link that carries the token) and drives cost-bearing external calls.
